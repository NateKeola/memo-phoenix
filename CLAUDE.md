# CLAUDE.md — Memo Phoenix

Single-user personal knowledge and companion system. Capture (voice memo, text, interview) feeds a miner that builds a personal knowledge graph, which a companion layer reads. The source of truth for scope and architecture is `docs/memo-phoenix-spec.md`. Read it before building. Build the V0 surface of a feature, then stop. V0 means exists and works, not polished.

## How to read the spec
- LOCKED items are constraints. Implement as written.
- OPEN-EXPLORATORY items: implement the recommended option and note the assumption in the PR description. Do not stall.
- If something contradicts a LOCKED item or is ambiguous at the architecture level, ask before building.

## Standing rules (every session)

### Data and security
- Single-user tenancy: `user_id` is the tenant key. RLS on every table, scoped to `user_id`. No exceptions.
- Service role key server-side only. Never in client code or the browser bundle.
- JWT validation on every authenticated route. Principle of least privilege everywhere.
- `captures`, `corrections`, `confirmations` are append-only. Every raw row carries `capture_id`. Every canonical row carries `source_claim_ids`. Provenance is mandatory.
- Canonical is recomputed from the full ground-truth set (raw + corrections + confirmations) on every run. Never edit canonical directly.
- Every canonical row carries a temporal class (evergreen | dated | decaying), a validity interval (`valid_from`, `valid_to`, `superseded_by`), plus `confidence`, `salience`, `last_confirmed_at`.
- Schema changes go through Supabase migration files only. No manual dashboard schema edits. Never apply an empty migration.

### Architecture
- The LLM is one stage of a deterministic pipeline, not the orchestrator. Routing goes in code wherever code can decide it.
- Hard-gate high-stakes actions (sending email, scheduling, anything with an external side effect) in code, not in a prompt instruction.
- Deterministic prompt composition only. No LLM meta-prompt.
- Direct Anthropic SDK. No LangChain.
- Cache system prompts (pad past 4096 tokens, mark ephemeral). Telemetry from day one: tool calls, miner runs, cache hit rate.
- Prompt template `.md` edits require regenerating and committing the `.generated.ts` files. Runtime filesystem reads fail in Vercel serverless.
- When a prompt is deprecated and replaced, move the old version into the local `reference/OldPrompts/` archive (gitignored; git history is the durable record), name the replacement with a new version suffix (e.g. `-v2-lean`), and repoint the generator at it. The active prompts dir holds only live prompts.

### Separation and credentials
- Memo Phoenix is its own world: own repo, own Supabase account or org, own Vercel project. There is no path to the Miine canonical Supabase project. Do not reference or connect to it.
- The GitHub token is a fine-grained PAT scoped to this repo only.
- Merge-driven model: the only privileged action from this environment is `gh` PR create and merge. Once wired, migrations apply via a GitHub Action on merge to main, and Vercel deploys via the Git integration. Platform tokens (Supabase, Vercel) live in GitHub Actions secrets, not in this environment. During bootstrap (PR0 and PR1) applying migrations directly to the dev project with `supabase db push` is acceptable.

### Writing
- No em dashes in any document or output.
- Direct, simple language. No formatting flourishes.

### Continuity
- After every substantive change (a shipped feature, a fix, an architecture or ops decision, a verified state), append a dated entry to the Decision log below capturing WHAT changed and WHY. Do it as part of the change, not later. This is how context survives compaction and handoffs without relying on session memory.
- Keep `docs/HANDOFF.md` current as part of the same change, not only at compaction time. The handoff document is a durable, continuously-updated record of the build state, deferred items, gotchas, and next steps. After each PR, update it so a fresh session can pick up from `docs/HANDOFF.md` plus this file alone.
- EVERY PR MUST update `docs/HANDOFF.md` in the same PR (a HANDOFF edit is part of the PR's diff, not a follow-up). Before opening or merging a PR: (a) add/refresh the build-state bullet for this branch, (b) bump the `Last updated:` line, (c) correct any statement this PR makes stale or false (do not leave contradictory prose, e.g. an old "X is failing" callout after X is fixed, or a "Y is configured" claim after finding Y is not). A PR that changes behavior or ops state but leaves HANDOFF untouched is incomplete. This applies to bug-fix PRs too, not just features.

- 2026-06-20: Deterministic-id hardening, the root-cause fix (branch `id-hardening`, PR open, NOT merged). INVESTIGATION: identity is `canonicalId = uuidv5(namespace, userId+table+normalizeLabel(label))` (`identity.ts`); people use `canonicalPersonId(first,last)` which reconstructs the label so it equals the label hash. Because the label is mutable, a label drift mints a NEW id, which is the single root cause of four symptoms. Exact drift points: (churn) `derive.ts` node/insight passes hash the model-reworded label -> new id -> `writeCanonical` INSERTs a near-duplicate (worst for phrase-labeled facts/insights; the live graph already has 13 normalized-label collisions like "todd"/"usc"/"interview agent" and 8 fuzzy drift-split clusters, found by the migration dry-run). (people-merge) `corrections.ts` computes loser/survivor ids by hashing labels and rewrites the label before the hash; works but is label-coupled and two genuinely-distinct same-name people collide. (companion overlay) `companion_state.commitment_id` = the commitment's label-hash id, so a commitment label drift orphans the overlay (mitigated only by the `matchOverlay` person+Jaccard re-match in `lib/companion/overlay.ts`). (freshness) `reconcileFreshness` keys `last_confirmed_at`/`salience` on the row id (`.eq('id', r.id)`) so a drift loses the decay anchor and a renewal lands on a new id; `supersedeFromDiscrepancies` keys the PLAN on claim_ids (drift-robust) but writes `superseded_by = survivorId` (a label-hash id that can drift). CHOSEN SCHEME: stable identity decoupled from the label via a persisted resolution layer. New `entity_aliases` table (migration 0014, FORCE RLS, per-user SELECT, service-role write, like miner_state) maps `(user_id, entity_table, alias_norm) -> stable_id`. The miner resolves an extracted entity through a deterministic conservative ladder (`resolution.ts` `resolveId`/`Resolver`): exact label -> known alias (persisted + the corrections rewrite) -> conservative fuzzy (token-Jaccard, default 0.8, env `MINER_RESOLVE_FUZZY`) with an optional CONTEXT KEY (commitments pass the linked person id: agreement relaxes the bar to 0.5, disagreement is a hard no that blocks even an exact-text match, mirroring the proven overlay re-match), with an ambiguity-margin guard, minting a RANDOM uuid only on no match. A label change records a new alias, never a new id. KEY SAFETY PROPERTY: existing canonical row ids ARE their stable ids (no re-key) and the resolver self-seeds from the current canonical rows (exact match adopts an existing id before minting), so turning the scheme on is non-destructive; bias is toward tightness because a missed match is no worse than today (mints, as today) while a false merge is the only regression, so the fuzzy tier is conservative and guarded. ALTERNATIVE CONSIDERED + REJECTED: keep deterministic ids but derive them from a more stable key (the provenance/claim set, or a content seed). Rejected because no mutable-content-derived key is stable: `source_claim_ids` GROWS as new mentions arrive (even less stable than the label), and free-text entities have no other intrinsic key. Identity must be REMEMBERED (persisted resolution state), not re-derived. (A second alternative, LLM-driven resolution at write time, violates the harness doctrine that routing/identity is deterministic code; the model only proposes aliases.) HOW THE FOUR SYMPTOMS RETIRE: churn -> a reworded long phrase resolves to its existing id (fuzzy + alias), so no duplicate INSERT (the worst phrase-labeled cases are exactly what fuzzy catches). people-merge -> exact/alias/correction resolve to one stable id and the alias union is durable; distinct same-name people are protected by the conservative fuzzy + (for commitments) the context key. companion-overlay drift -> the commitment id is now stable across a label drift, so the overlay keys on it and never orphans (the `matchOverlay` fuzzy fallback becomes a belt, not the load-bearing path). freshness drift -> the decay anchor + salience + `superseded_by` all key on a stable id that survives drift, so the aging row keeps its history and a renewal lands on the same id. PRECISELY SCOPED PARTIAL: a VERY short commitment relabel with low token overlap AND no/!= person (e.g. "call" -> "ping" with the person unknown) still mints; that is no worse than today and the overlay person+fuzzy layer still re-associates the state.
